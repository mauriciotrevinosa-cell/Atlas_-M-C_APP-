## 4пёҸвғЈ Decisiones ArquitectГіnicas Tomadas

### рҹ”· ADR-001: Arquitectura Multi-Capa (2026-01-30)
**DecisiГіn:** SeparaciГіn en 3 capas principales (UI / Services / Core)  
**RazГіn:** Modularidad, testing aislado, escalabilidad  
**Trade-offs:** MГЎs complejidad inicial, mejor mantenibilidad long-term  
**Estado:** вң… Implementado en estructura de carpetas

### рҹ”· ADR-002: Data Layer con MГәltiples Proveedores (2026-01-30)
**DecisiГіn:** Abstract base class `DataProvider`, mГәltiples implementaciones  
**RazГіn:** Flexibilidad para cambiar/agregar fuentes de datos  
**Trade-offs:** Overhead de abstracciГіn vs vendor lock-in  
**Estado:** вң… Implementado (Yahoo Finance operativo)

### рҹ”· ADR-003: ARIA 100% AutГіnoma (2026-02-01)
**DecisiГіn:** Ollama (local, open source) como backend primario, no APIs comerciales  
**RazГіn:** Control total, privacidad, costo $0, sin vendor lock-in  
**Trade-offs:** Modelos menos potentes sin GPU, pero escalable con hardware  
**Estado:** вҸі En implementaciГіn (modelos descargando)

### рҹ”· ADR-004: Multi-Cerebro System para ARIA (2026-02-01)
**DecisiГіn:** Arquitectura con cerebro maestro + cerebros especializados  
**RazГіn:** Modularidad, mejor calidad por tarea, escalabilidad  
**Trade-offs:** Complejidad de orquestaciГіn vs monolito simple  
**Estado:** рҹ”„ DiseГұado, pendiente implementaciГіn

### рҹ”· ADR-005: Probabilistic vs Deterministic Approach (2026-01-30)
**DecisiГіn:** Atlas es probabilГӯstico, no determinista  
**RazГіn:** Mercados son inherentemente probabilГӯsticos  
**Trade-offs:** MГЎs complejo de explicar, mГЎs realista  
**Estado:** вң… Principio arquitectГіnico adoptado

### рҹ”· ADR-006: Python como lenguaje primario (2026-01-30)
**DecisiГіn:** Python para desarrollo inicial, C++ reservado para HFT futuro  
**RazГіn:** Velocidad de desarrollo, ecosistema ML/quant, hiring  
**Trade-offs:** Performance vs productividad (resolver con C++ si/cuando se necesite)  
**Estado:** вң… Implementado

### рҹ”· Decisiones sin ADR formal (por simplificar):
- Licencia Proprietary (M&C)
- Estructura de docs/ modular por fase
- `.env` para configuraciГіn sensible
- `pyproject.toml` para gestiГіn de dependencias

**Nota:** No estamos sobre-documentando. ADRs solo para decisiones arquitectГіnicas mayores que afectan 3+ mГіdulos o son irreversibles/costosas de cambiar.

## 5пёҸвғЈ Riesgos y Deuda TГ©cnica (Top 10)

### рҹ”ҙ **ALTO RIESGO:**

**1. ARIA Ollama sin GPU potente (actualidad)**  
- **Riesgo:** Modelos grandes (Mixtral) serГЎn muy lentos en hardware actual  
- **Impacto:** ARIA limitada a modelos pequeГұos hasta tener eGPU  
- **MitigaciГіn:** Usar modelos ligeros en FASE 1 (llama3.1:8b, qwen2.5:7b), upgrade en FASE 2  
- **Timeline:** Resuelto en 3-6 meses con eGPU

**2. Falta de testing automatizado**  
- **Riesgo:** Regresiones no detectadas, refactors peligrosos  
- **Impacto:** Bugs en producciГіn, tiempo perdido debuggeando  
- **MitigaciГіn:** Agregar pytest + CI/CD en siguiente sprint  
- **Timeline:** 1-2 semanas

**3. Sin backtesting engine implementado**  
- **Riesgo:** No podemos validar estrategias, Atlas no es usable para trading real  
- **Impacto:** Proyecto no genera valor hasta FASE 11  
- **MitigaciГіn:** Priorizar backtesting en roadmap (FASE 11 adelantada si es posible)  
- **Timeline:** 4-6 semanas de desarrollo

### рҹҹЎ **RIESGO MEDIO:**

**4. DocumentaciГіn extensa pero cГіdigo limitado**  
- **Riesgo:** Docs se desactualizan, overhead de mantenimiento  
- **Impacto:** ConfusiГіn, tiempo perdido sincronizando docs  
- **MitigaciГіn:** Docs reflejan diseГұo, cГіdigo sigue. Actualizar docs cada sprint.  
- **Timeline:** Ongoing

**5. ARIA Multi-Cerebro no implementado**  
- **Riesgo:** ARIA limitada a conversaciГіn simple, no usa especializaciГіn  
- **Impacto:** ARIA menos Гәtil de lo planeado  
- **MitigaciГіn:** Implementar en siguiente sprint (FASE 1.5)  
- **Timeline:** 1 semana

**6. Falta de cache en Data Layer**  
- **Riesgo:** Descargas repetidas de datos, lentitud, rate limits  
- **Impacto:** Desarrollo lento, posibles bans de APIs  
- **MitigaciГіn:** Implementar cache simple (CSV) en Data Layer  
- **Timeline:** 2-3 dГӯas

**7. Sin CI/CD pipeline**  
- **Riesgo:** Deployments manuales, propensos a error  
- **Impacto:** Bugs en producciГіn, rollbacks manuales  
- **MitigaciГіn:** Setup bГЎsico de GitHub Actions  
- **Timeline:** 1 dГӯa

### рҹҹў **RIESGO BAJO (pero importante):**

**8. UI web no implementada**  
- **Riesgo:** Atlas solo usable por CLI/cГіdigo  
- **Impacto:** DifГӯcil de demostrar, menos user-friendly  
- **MitigaciГіn:** Implementar UI bГЎsica en FASE 12  
- **Timeline:** 3-4 semanas (post-MVP)

**9. C++ module reservado pero vacГӯo**  
- **Riesgo:** Si HFT se vuelve crГӯtico, delay por empezar de cero  
- **Impacto:** Oportunidad de mercado perdida  
- **MitigaciГіn:** EstГЎ OK por ahora, focus en Python MVP  
- **Timeline:** 1+ aГұo (FASE 16-17)

**10. Derivatives dashboard planeado pero no iniciado**  
- **Riesgo:** Funcionalidad Гәtil (liquidation heatmaps) no disponible  
- **Impacto:** Missing alpha en trading  
- **MitigaciГіn:** Implementar post-MVP, prioridad media  
- **Timeline:** 2-3 semanas (post-core)
